    GRASP

    GRASP (General Responsibility Assignment Software Patterns) is a set of principles that help assign responsibilities to classes and objects in object-oriented design.

    These patterns answer the question:
    "Who should be responsible for performing a particular task?"

    GRASP helps build clean, scalable, and flexible architectures, especially in the early stages of design.

    GRASP Patterns

    # Pattern Name What It Does
    1 Information Expert Assign responsibility to the class that has the needed information
    2 Creator Defines who should create an object
    3 Controller Delegates handling of user input
    4 Low Coupling Minimizes dependencies between classes
    5 High Cohesion Maximizes related functionality within a class
    6 Polymorphism Delegates behavior via a common interface
    7 Pure Fabrication Introduces a helper class to reduce coupling
    8 Indirection Introduces a mediator to weaken dependencies
    9 Protected Variations Protects parts of the system from changes by using stable interfaces

    Examples (Kotlin-style)

    1. Information Expert

    The class with the necessary information is responsible. 

    class Order(val items: List<Item>) {
    fun getTotalPrice(): Double {
        return items.sumOf { it.price }
    }
}

    Order is the expert of its total because it contains the list of items.

    2. Creator

    Who should create the object? Someone who:

    • Aggregates it
    • Uses it
    • Has the necessary data
    class Order {
    fun addItem(product: Product) {
        val item = Item(product)
        // Add to order
    }
}

    Order creates Item because it owns it.

    3. Controller

    Assigns the object responsible for handling user input.

    class OrderController {
    fun onOrderButtonClicked() {
        // calls OrderService or similar
    }
}

    OrderController manages the event, not the UI.

    4. Low Coupling

    Minimize dependencies between classes.

    Instead of Order directly depending on Database, depend on an abstraction like OrderRepository.

    5. High Cohesion

    Each class does one thing well. If a class handles too many responsibilities, cohesion decreases.

    6. Polymorphism

    Shared behavior through an interface.

    interface PaymentMethod {
    fun pay(amount: Double)
}

class CreditCard : PaymentMethod { ... }
class PayPal : PaymentMethod { ... }

    7. Pure Fabrication

    A class created not for the domain, but for architectural purposes.

    class Logger {
    fun log(message: String) { ... }
}

    Logger is not part of the domain but helps reduce coupling.

    8. Indirection

    A mediator to reduce direct dependencies.

    class PaymentService {
    fun process(method: PaymentMethod, amount: Double) {
        method.pay(amount)
    }
}

    Instead of calling CreditCard.pay() directly, we use a mediator.

    9. Protected Variations

    Changes in one part should not break others.

    Example: an interface Storage with implementations DatabaseStorage and FileStorage.

    Conclusion

    GRASP helps:

    • Make code readable and maintainable
    • Reduce coupling
    • Increase reuse and modularity
